# 第一阶段样例视频输入处理实现计划

## 1. 阶段目标

现状：`Apps/Workbench` 已有静态工作台 demo，五区布局已成型，JS/CSS 已拆分，单文件未超过 200 行；但当前只做本地预览和模拟派生产物，没有真实后端、真实 FFmpeg、持久化、自动测试和正式契约。

第一阶段目标：保留现有类剪映工作台壳，把“样例视频输入与处理”从前端模拟升级为真实主链路。

```text
上传样例视频
-> 后端保存
-> FFmpeg/ffprobe 处理
-> 生成 SampleVideoArtifact
-> 前端展示预览、元信息、抽帧时间线
```

本阶段不做视频理解、结构拆解、结构迁移真实模型调用和正式导出。

## 2. 前端实现

保留当前 `Apps/Workbench` 五区布局，不重写成 React。

需要实现：

- 左侧素材区接入真实上传状态，不再只依赖 `URL.createObjectURL` 生成模拟结果。
- 中央预览区使用后端返回的标准化视频或原视频访问地址。
- 底部时间线使用后端返回的抽帧列表。
- 每个抽帧必须带 `timestamp`、`artifactId`、`parentArtifactId`。
- 右侧属性区展示真实处理状态、视频元信息、错误摘要和 trace 信息。
- 当前模拟的“进入视频理解”“生成迁移方案”保留为占位，不接真实 AI。

前端只负责 UI、交互、状态展示和调用 API，不直接实现媒体处理、存储或业务核心判断。

## 3. API 与后台处理

新增真实上传入口：接收一个样例视频，创建 `Workspace` 或挂到当前默认 workspace。

上传后立即返回：

- `processingJobId`
- `sampleVideoId`
- `traceId`

前端通过任务状态接口轮询处理进度。

后台处理链路固定为：

```text
保存原始视频
-> 校验格式/大小/时长
-> ffprobe 提取元信息
-> 生成封面帧
-> 固定间隔抽帧
-> 抽取音频轨
-> 写入 SampleVideoArtifact metadata
-> 标记 processed / failed
```

第一版使用本地文件系统和 JSON metadata 存储。实际视频、帧图、音频放入 `Runtime`，不入库。

HTTP 上传请求不等待完整 FFmpeg 处理完成。

## 4. 核心契约

第一版至少定义并稳定这些对象。

### TraceContext

用于贯穿一次处理链路。

必须包含：

- `runId`
- `traceId`
- `stageId`

### ArtifactRef

用于引用核心产物。

必须包含：

- `artifactId`
- `parentArtifactId`
- `type`
- `uri`
- `summary`

### SampleVideoArtifact

用于表示处理完成后可被后续视频理解消费的样例视频产物。

必须包含：

- 原始视频引用。
- 标准化视频引用。
- 封面帧。
- 抽帧列表。
- 音频轨。
- 视频元信息。
- 处理状态。
- trace 信息。

### FrameArtifact

用于表示一张抽帧。

必须包含：

- `frameId`
- `artifactId`
- `parentArtifactId`
- `timestamp`
- `imageUri`

### ProcessingJob

用于表示后台处理任务。

必须包含：

- `jobId`
- `stage`
- `status`
- `progress`
- `errorSummary`

硬要求：

- 阶段输出不能是裸文本。
- 派生产物必须有 `parentArtifactId`。
- 所有处理日志必须能通过 `traceId` 串起来。
- 失败必须保留错误摘要和 DebugSnapshot 引用。

## 5. 模块边界

- `Apps/Workbench`：UI、交互、状态展示和调用 API。
- `Apps/Api`：上传、状态查询、artifact 查询和任务触发。
- `Infrastructure/MediaProcessing`：ffprobe、抽帧、封面、抽音频。
- `Infrastructure/Storage`：文件与 metadata 存取。
- `Infrastructure/Observability`：stage 日志和 DebugSnapshot。
- `Core/Workspace`：workspace 与 sample artifact 的关系。
- `Core/SampleUnderstanding`：本阶段只接收处理完成的样例资产，不做理解实现。

文件粒度要求：

- 单业务文件原则上不超过 200 行。
- 超过 300 行必须拆分。
- UI、媒体处理、存储、观测不能堆在同一个文件。
- 核心流程文件只负责编排，不直接实现具体 FFmpeg、存储或日志细节。

## 6. API 设计

### 上传样例视频

```text
POST /api/workspaces/:workspaceId/sample-videos
```

输入：视频文件。

输出：

- `processingJobId`
- `sampleVideoId`
- `traceId`

### 查询处理任务

```text
GET /api/processing-jobs/:jobId
```

输出：

- 状态。
- 进度。
- 当前 stage。
- 错误摘要。

### 查询样例视频产物

```text
GET /api/sample-videos/:sampleVideoId/artifact
```

输出：完整 `SampleVideoArtifact`。

### 访问本地运行资源

```text
GET /runtime/...
```

输出：本地开发期可访问的视频、封面、抽帧、音频资源。

## 7. 错误状态

错误状态固定为：

- `invalid_file_type`
- `file_too_large`
- `duration_too_long`
- `metadata_probe_failed`
- `frame_extract_failed`
- `audio_extract_failed`
- `storage_write_failed`

前端只展示用户可理解的错误摘要，不展示裸 stack、完整本地路径或敏感内容。

## 8. 测试设计

自动化测试只覆盖代码和模块，不做大量 UI 功能测试。

### Unit

必须覆盖：

- 文件校验规则。
- 抽帧时间戳策略。
- artifact parent 关系。
- processing job 状态流转。
- trace 字段创建和传递。

### Contract

必须覆盖：

- `SampleVideoArtifact` 必填字段完整。
- `FrameArtifact` 必须带 `timestamp` 和 `parentArtifactId`。
- 阶段输出不能缺少 `runId / traceId / stageId / artifactId`。

### Integration

必须覆盖：

- 使用一个极短 mp4 跑真实 ffprobe、封面、抽帧、抽音频。
- 损坏视频返回结构化失败。
- 无音频视频不阻塞封面和抽帧生成。

### 手动验收

必须验证：

- 打开工作台，上传短视频，看到处理状态、封面、元信息、抽帧时间线。
- 点击抽帧，预览区能跳转到对应时间。
- 错误视频能显示友好错误，不出现裸 stack 或完整敏感路径。

## 9. 任务卡

### 任务 1：建立 API 与任务状态骨架

目标：实现上传入口、任务状态查询和样例 artifact 查询的基础路由。

输入：

- workspaceId。
- 视频文件。

输出：

- processingJobId。
- sampleVideoId。
- traceId。

验收条件：

- 上传接口立即返回任务信息。
- 状态接口能返回 pending、processing、processed、failed。
- artifact 查询在未完成时返回明确状态。

不做：

- 不做真实视频理解。
- 不做模型调用。

### 任务 2：实现本地存储与 metadata

目标：把上传文件、派生产物和 metadata 存入 `Runtime`。

输入：

- 原始视频文件。
- 派生产物。
- SampleVideoArtifact metadata。

输出：

- 可被 API 返回的 artifact metadata。
- 可被前端访问的本地资源 uri。

验收条件：

- 运行产物不进入 git。
- metadata 能描述原视频、封面、抽帧、音频和父子关系。

不做：

- 不做云存储。
- 不做用户权限。

### 任务 3：实现媒体处理链路

目标：用 FFmpeg/ffprobe 完成视频基础处理。

输入：

- 原始视频路径。
- TraceContext。

输出：

- 视频元信息。
- 封面帧。
- 抽帧列表。
- 音频轨。

验收条件：

- 每个派生产物都有 artifactId 和 parentArtifactId。
- 抽帧带 timestamp。
- 失败返回结构化错误。

不做：

- 不做 OCR。
- 不做 ASR。
- 不做镜头切分。

### 任务 4：前端接入真实处理状态

目标：把现有工作台从模拟数据改为读取 API 数据。

输入：

- 上传文件。
- processingJob 状态。
- SampleVideoArtifact。

输出：

- 处理状态展示。
- 预览区视频。
- 右侧元信息。
- 底部抽帧时间线。

验收条件：

- 上传后能看到处理进度。
- 处理完成后能看到真实封面、元信息和抽帧。
- 点击抽帧能跳转视频时间。

不做：

- 不重写整体 UI。
- 不接真实视频理解。

### 任务 5：补齐模块测试

目标：建立第一阶段必要的 unit、contract、integration 测试。

输入：

- 测试视频 fixture。
- mock metadata。
- 真实媒体处理函数。

输出：

- 可重复运行的模块测试。

验收条件：

- 默认测试覆盖 unit + contract。
- 媒体集成测试可单独运行。
- 失败用例能验证结构化错误。

不做：

- 不做大量 UI E2E。
- 不调用真实模型。

## 10. 默认决策

- 当前静态工作台作为第一版 UI 基础保留，不立即迁移到 React。
- 第一阶段只做“样例视频输入与处理”的真实化。
- 视频理解按钮继续保留，但接收真实 `SampleVideoArtifact` 后再进入下一阶段。
- 本地开发优先，暂不做云存储、用户登录、权限、多租户和正式部署。
- FFmpeg/ffprobe 是媒体处理默认工具。
- 模型调用、ASR、OCR、镜头切分、结构拆解都放到第二阶段。
