## 架构风险

- 可理解性：样例处理链路 stage 清楚；切镜链路已经拆出输入准备、联表、Agent、校验、修复、合并，但 `enqueue` 同时承担缓存预判、联表生成和启动 job，入口职责偏重。
- 可追踪性：主链路总体具备 `runId / traceId / stageId / artifactId / parentArtifactId`；切镜历史里的 `traceId` 仍指向样例处理 trace，不足以区分切镜刷新、复用和重跑。
- 返工/历史：`shotBoundaryAnalysisHistory` 会追加历史，不会静默覆盖；但历史条目的来源 trace 不准确，影响后续对比和返工定位。
- 耦合风险：切镜入口、缓存、联表生成、Agent 调用、历史写入集中在一个 800 行以上文件，后续变化点容易相互牵连。
- 演进风险：`Apps/Api/lib/shot-boundary-service.js`、`Apps/Api/lib/shot-boundary-analysis.js`、`Apps/Api/lib/sample-processing-service.js` 等核心文件超过 500 行，需要后续拆分或留下明确豁免理由。

## 性能风险概览

- 最可能卡顿的交互：用户点击切镜分析后，HTTP 请求在返回 `202` 前同步生成 contact sheet，并做缓存查找；长视频或大帧图会让前端看起来像提交卡住。
- 主线程风险：前端音频波形已优先走 Worker，但 Worker 失败后会在主线程重新 `fetch + arrayBuffer + decodeAudioData`，大音频下仍可能短时阻塞。
- 重渲染风险：播放进度已经主要通过 DOM transform 和 RAF 更新，未发现明显全局 reducer 高频更新。
- 布局/绘制风险：时间线波形使用静态 canvas 缓存，风险可接受。
- 资源生命周期风险：波形 Worker、RAF、AudioContext 基本有释放路径；主线程 fallback 没有 AbortController，切换音频后旧 fetch 只能靠 requestId 防止提交，不能取消网络和解码成本。
- UI 设计风险：切镜提交前的同步准备没有 job 进度，失败也不会返回切镜 job，用户只能看到顶层请求失败。

## 可观测性结论

- 本次变更是否影响主链路：本轮无代码变更；检查发现现有切镜入口的前置准备没有纳入切镜 stage。
- 本次变更是否影响日志/调试：本轮新增本基线文档；发现 `shot.cache_reuse` miss 路径缺 `stage.start`。

## 阻塞问题

1. P1：切镜 `enqueue` 在创建 job 和写 `shot.*` stage 前同步生成 contact sheet。
   - 文件位置：`Apps/Api/lib/shot-boundary-service.js:56`
   - 触发条件：用户发起切镜分析，特别是帧数多、图片大、Sharp 处理慢或联表生成失败时。
   - 用户影响：请求返回 `202` 前卡住；失败只会落入顶层 `api.request.handle`，没有切镜 job、进度、stage fail 和切镜 DebugSnapshot。
   - 建议方向：`enqueue` 只做轻量校验、创建 job、返回 `202`；缓存探测和 contact sheet 生成进入 `runAnalysis` 的 `shot.input_prepare / shot.contact_sheet / shot.cache_reuse` stage。

2. P1：切镜历史记录使用样例处理 trace，而不是本次切镜 trace。
   - 文件位置：`Apps/Api/lib/shot-boundary-service.js:574`
   - 触发条件：切镜成功、失败、刷新、缓存复用都会追加 `shotBoundaryAnalysisHistory`。
   - 用户影响：历史列表无法准确追溯某次切镜运行；刷新和复用会看起来来源相同，后续返工、对比、Debug 定位困难。
   - 建议方向：`attachAnalysis` 接收当前 analysis trace 或从 analysis/agent 上读取本次 trace；history 同时保留 `traceId`、`sourceTraceId`、`turnId`、`resultOrigin`。

## 风险问题

1. P2：缓存 miss 日志只有 `stage.end`，没有 `stage.start`。
   - 文件位置：`Apps/Api/lib/shot-boundary-service.js:627`
   - 风险：违反核心 stage 必须有结构化开始、结束、失败日志的约束；后续审计时无法还原 cache lookup 输入。
   - 建议方向：把 cache lookup 包成完整 `runStage`，miss、hit、eligibility rejected 都记录 start/end，失败记录 fail 和 DebugSnapshot。

2. P2：多个核心文件超过架构约束中的 500 行软上限。
   - 文件位置：`Apps/Api/lib/shot-boundary-service.js` 852 行；`Apps/Api/lib/shot-boundary-analysis.js` 775 行；`Apps/Api/lib/sample-processing-service.js` 618 行；`Apps/Workbench/src/components/WorkbenchApp.tsx` 540 行；`Apps/Workbench/src/components/PropertyPanel.tsx` 520 行。
   - 风险：入口编排、领域归一、缓存、模型调用、UI 状态继续堆叠后，局部修改容易牵连多条链路。
   - 建议方向：优先拆切镜 service 的 job orchestration、cache lookup、history write；拆 analysis 的 prompt/parse/validation/cache params；UI 文件只拆有稳定边界的 hook 或 panel helper。

3. P2：音频波形 Worker 失败后主线程 fallback 不能取消。
   - 文件位置：`Apps/Workbench/src/hooks/useAudioWaveform.ts:146`
   - 风险：切换音频或离开页面后，旧的 fetch/decode 仍可能消耗网络、内存和主线程预算；大音频下影响交互。
   - 建议方向：给 worker 和主线程 fallback 都加 AbortController 或等价取消机制；旧请求取消后不再继续 decode。

- 目标：消除切镜提交前同步重型处理，让 contact sheet、cache lookup 和失败都进入可追踪 job/stage。
- 文件范围：`Apps/Api/lib/shot-boundary-service.js`、相关 `Tests/unit/threadpool-shot-boundary.test.js`。
- 验收条件：
  - `POST /api/sample-videos/:id/shot-boundary` 在非 cache-hit 路径只做轻量校验、创建 job、返回 `202`。
  - contact sheet 生成失败写入 `shot.contact_sheet` 的 `stage.fail` 和 DebugSnapshot。
  - cache lookup miss/hit/eligibility rejected 都有完整 start/end。

### 任务卡 2：修正切镜历史 trace 血缘

- 目标：让每条 `shotBoundaryAnalysisHistory` 能追到本次切镜运行、来源样例和 Agent turn。
- 文件范围：`Apps/Api/lib/shot-boundary-service.js`、`Apps/Api/lib/shot-boundary-analysis.js`、`Infrastructure/ArtifactIndex/artifact-index.js`、相关 history/cache 测试。
- 验收条件：
  - 新历史条目包含本次切镜 `traceId`。
  - 如需保留样例处理来源，使用 `sourceTraceId` 或 `sampleTraceId`。
  - cache reuse history 能区分复用来源 `sourceSampleVideoId/sourceTurnId` 和当前运行 trace。
  - 旧 artifact 缺字段时前端仍能兼容展示。
- 不做的事：不迁移历史 runtime 数据，不删除旧 history 字段。

### 任务卡 3：为超长核心文件建立拆分计划

  - 优先拆出已有稳定边界，不改变外部 API。
  - 拆分后关键测试仍通过。
- 不做的事：不为了行数做纯搬运式大重构，不顺手改业务行为。

### 任务卡 4：补齐音频波形 fallback 取消机制

- 目标：避免旧音频解码请求在切换后继续消耗资源。
- 文件范围：`Apps/Workbench/src/hooks/useAudioWaveform.ts`、`Apps/Workbench/src/workers/audioPeaks.worker.ts`、前端 trace/performance 测试。
- 验收条件：
  - URL 切换或组件卸载时，旧 fetch/decode 有取消路径或明确停止继续处理。
  - 取消不写 `stage.fail`，真实解码失败仍写 `stage.fail` 与 DebugSnapshot。
  - 快速切换音频不会让旧请求覆盖新波形。
- 不做的事：不改变波形视觉样式，不把整段音频数据写入日志。

## 通过标准

- 核心结果能追踪到准确的用户操作、stage、artifact、父 artifact 和模型 turn。
- 切镜重跑、刷新、缓存复用不会静默覆盖历史，并能区分当前运行 trace 与来源样例 trace。
- 每个核心 stage 至少具备 start/end/fail 的可还原结构化日志。
- 重型媒体准备不阻塞用户提交请求，不把失败留在顶层 request trace 里。
- 高频前端交互不触发全局重渲染，大文件解码和旧异步任务有取消或释放路径。
- 超过 500 行的核心文件有拆分计划或明确豁免理由。
