# 基础架构约束

## 定位

本文档是爆款结构迁移引擎的长期架构约束。它约束后续代码、模块、数据流、返工、版本、调试和审查方式。

`PlanMD` 只用于阶段计划和讨论结果；长期工程约束以 `Docs/Architecture` 下的文档为准。

## 核心概念

- `Project`：用户的一次创作工作区，承载样例、用户输入、阶段产物、版本和分支。
- `Stage`：主链路中的一个处理阶段，例如样例理解、结构拆解、结构迁移、方案生成、结果展示。
- `Artifact`：某个阶段生成的不可变产物。核心阶段结果必须 artifact 化，不能只存在于临时变量里。
- `Version`：同一阶段或同一分支下的产物版本。返工、重跑、策略切换都应产生新版本。
- `Branch`：从某个历史 artifact 开出的新尝试，用于支持不同迁移方式、不同生成方向和方案对比。
- `Trace`：一次运行链路的追踪信息，用于串联 run、stage、artifact、日志和调试快照。

## 模块边界

- 样例理解模块只负责建立样例视频的基础理解，不生成用户新方案。
- 结构拆解模块只负责解释样例为什么成立，并输出可用于迁移的结构分析。
- 结构迁移模块只负责把样例结构映射到用户新内容，不负责最终展示和润色。
- 方案生成模块只基于迁移结果生成脚本、分镜、字幕、标题等方案内容，不回写上游拆解。
- 结果展示模块只消费领域产物并生成展示视图，不直接调用模型或修改业务产物。
- 工作流模块负责阶段编排、版本、分支、返工和 artifact 血缘，不承载具体业务推理。
- 模型调用模块统一管理模型入口、prompt/template 版本、输入输出解析和失败处理。
- 可观测性模块统一管理日志、trace、debug snapshot 和错误定位，不由各模块自造一套。

## 数据流规则

主链路必须通过阶段产物流转：

```txt
SourceVideo
-> VideoUnderstanding
-> StructureAnalysis
-> TransferRuleSet / TransferMapping
-> GeneratedPlan
-> ResultViewModel
```

约束：

- 模块之间只通过明确输入和输出产物通信。
- 禁止共享一个可被所有模块读写的巨型 context。
- 下游模块不得直接读取上游模块内部状态、临时变量或模型原始返回。
- 展示层不得直接消费模型原始文本，必须消费经过领域对象或展示模型整理后的结果。
- 运行上下文、日志上下文和业务产物分开管理，通过 `traceId` 和 `artifactId` 关联。

## Artifact、版本和返工规则

- 核心阶段产物默认不可覆盖，只能新增版本。
- 用户返工、重跑、切换策略、创建分支时，必须保留来源 artifact。
- 新 artifact 必须能追溯到它的上游 artifact、生成策略、schema 版本和 trace。
- 废弃结果应标记状态，不应物理删除或静默覆盖。
- 多方案对比必须基于不同 artifact 或 branch，不允许用同一个可变对象反复改写。

## 变化承接规则

高频变化不应散落在业务代码中，优先使用以下通道承接：

- 策略：迁移方式、生成方式、评估方式等可替换逻辑。
- 配置：结构维度、输出模块、平台规则、展示开关等可调整选项。
- `schemaVersion`：阶段产物和 payload 的结构版本。
- payload 扩展：变化快的业务字段放在阶段产物 payload 中，外层 artifact 结构保持稳定。

新增核心功能前必须回答：

- 它属于哪个核心概念或阶段？
- 它改变的是流程、策略、配置、schema，还是展示？
- 它是否影响历史 artifact 的读取和复现？
- 它是否需要新的审查规则或可观测性字段？

## 禁止事项

- 禁止展示层直接调用模型。
- 禁止模型调用散落在多个业务模块中。
- 禁止下游模块修改上游 artifact。
- 禁止绕过 artifact 直接传递核心阶段结果。
- 禁止把返工实现成覆盖旧结果。
- 禁止所有模块混用同一套无语义日志。
- 禁止在普通日志中裸写敏感内容、完整 prompt 或完整素材内容。
- 禁止新增无法归入核心概念的模块而不先更新架构约束。

## 审查约束

- 架构边界、领域概念、artifact、版本、分支和数据流由 `viral-structure-architecture-reviewer` 检查。
- 日志、调试、trace、debug snapshot 和模型调用留痕由 `viral-structure-observability-reviewer` 检查。
- 每轮核心功能交付前，应至少进行一次 diff review。
- 改动核心领域对象、workflow、artifact、branch、version、模型统一入口或可观测性基础设施时，必须升级为 full review。
