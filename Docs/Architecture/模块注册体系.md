# 模块注册体系

## 定位

模块注册体系是后端权威能力目录，用于描述可独立运行、可追踪、可重跑、可产出核心结果的能力单元。

模块不是代码目录，也不是页面、workflow 或执行器。模块注册只回答能力如何被系统调用、依赖什么、产出什么、如何追踪和如何暴露安全 metadata。

## 边界

- `Module`：能力产物粒度的单元，例如脚本段落分析、节奏结构分析、包装结构分析、未来生图、视频生成、结构迁移生成。
- `Executor`：模块的执行方式，例如 `threadpool-role`、本地 service、外部 API、AppServer conversation。`ThreadPool` 和 AppServer 本身不是模块。
- `Workflow`：编排多个模块和专用步骤的图或流程，例如完整分析。workflow 消费模块 descriptor，不拥有模块内部契约。
- `Surface`：前端页面、按钮、panel、对话框等展示绑定。前端只消费后端安全投影，不另建权威注册表。

## Descriptor 要求

每个正式模块至少说明：

- `moduleId`：稳定能力 ID。
- `moduleKind`：能力类型，例如 `structure-analysis`。
- `executorKind`：执行器类型。
- `dependencies`：上游 artifact 依赖。
- `artifact`：结果 key、history key、artifact type。
- `cacheKind`：可缓存模块的缓存类型；不可缓存填 `null`。
- `stages`：核心 stage 名称。
- `ui`：允许前端消费的安全展示 metadata。
- `startOptionsFromBody`：API body 到启动参数的转换。
- `createService`：创建实际 executor service。
- `getArtifact` / `buildCacheParams`：artifact 读取和缓存参数构造。

不得暴露 `skillPath`、函数引用、service 实例、本地路径或 executor 内部细节到公开 API。

## 新增模块标准

新增模块时优先新增 descriptor，并确认：

- 是否真的需要独立 artifact、cache、重跑、trace 或独立 executor。
- 是否能明确输入、输出、依赖和 parent artifact。
- 是否有 `stage.start / stage.end / stage.fail` 和 DebugSnapshot。
- 是否保留历史，不静默覆盖核心结果。
- 是否只向前端暴露安全 metadata。
- workflow 是否只依赖 descriptor，而不是写死模块实现。
