# 模块注册体系

## 定位

模块注册体系是后端权威能力目录，用于描述可独立运行、可追踪、可重跑、可产出核心结果的能力单元。

模块不是代码目录，也不是页面、workflow 或执行器。模块注册只回答能力如何被系统调用、依赖什么、产出什么、如何追踪和如何暴露安全 metadata。

## 边界

- `Module`：能力产物粒度的单元，例如脚本段落分析、节奏结构分析、包装结构分析、未来生图、视频生成、结构迁移生成。
- `Executor`：模块的执行方式，例如 `threadpool-role`、本地 service、外部 API、AppServer conversation。`ThreadPool` 和 AppServer 本身不是模块。
- `Workflow`：编排多个模块和专用步骤的图或流程，例如完整分析。workflow 消费模块 descriptor，不拥有模块内部契约。
- `Surface`：前端页面、按钮、panel、对话框等展示绑定。前端只消费后端安全投影，不另建权威注册表。

## Executor Registry

Executor Registry 是内部执行器目录，负责把“怎么执行”从模块 descriptor 和 workflow runtime 中抽出来。模块只声明 `executorKind` 和内部 `executorRef`，不直接持有 AppServer、ThreadPool、HTTP API 的生命周期细节。

当前执行器：

- `local-service`：调用本地 service 方法，例如样例导入。
- `threadpool-role`：调用现有 role analysis service，底层可以继续使用 ThreadPool。
- `appserver-turn`：封装 AppServer thread/turn 生命周期，只负责 start thread、submit turn、collect turn、timeout/retry 语义、safe error 和 stage log。
- `external-api`：预留远程 HTTP API 契约，未接业务。
- `remote-job`：预留异步远程任务契约，未接业务。

`appserver-turn` 的边界很窄：它返回 `threadId / turnId / status / finalMessage / activeThreadMessage / errorSummary` 等执行摘要，不写 artifact、不做 cache、不做 shot/script/rhythm/packaging 的业务校验，也不知道模块领域结构。

新增 executor 时必须确认：

- 是否是可被多个模块复用的执行方式，而不是某个模块的业务步骤。
- 是否统一返回结构化 `ExecutionResult`，并包含安全摘要和外部运行引用。
- 是否通过 `stage.start / stage.end / stage.fail` 写入 trace，失败时能生成受控 DebugSnapshot。
- 是否避免在普通日志、公开 API 或安全 metadata 中暴露完整本地路径、prompt 全文、原始模型输出或敏感内容。

## Workflow Descriptor

Workflow Descriptor 是 workflow 的后端权威编排声明。完整分析是 workflow，不是模块；它通过 descriptor 描述节点、顺序、并行组、可重跑节点、阻塞节点和聚合节点。

`full-analysis` 当前节点：

- `upload`：`sample-ingest` 模块节点，阻塞节点。
- `shotBoundary`：`shot-boundary` 模块节点，依赖上传结果，阻塞且可重跑。
- `scriptSegment` / `rhythmStructure` / `packagingStructure`：结构分析并行组，依赖切镜结果，可重跑。
- `aggregate`：内置聚合节点，只汇总当前 artifact 状态。

Workflow runtime 只按 descriptor 解析模块节点，并通过 module registry 调用模块；不直接维护结构分析模块 ID 列表。新增同类结构分析模块时，优先只改 workflow descriptor 和对应 module descriptor。

## Descriptor 要求

每个正式模块至少说明：

- `moduleId`：稳定能力 ID。
- `moduleKind`：能力类型，例如 `structure-analysis`。
- `executorKind`：执行器类型。
- `executorRef`：内部执行器引用，可包含非公开执行配置；不得进入公开投影。
- `dependencies`：上游 artifact 依赖。
- `artifact`：结果 key、history key、artifact type。
- `cacheKind`：可缓存模块的缓存类型；不可缓存填 `null`。
- `stages`：核心 stage 名称。
- `ui`：允许前端消费的安全展示 metadata。
- `startOptionsFromBody`：API body 到启动参数的转换。
- `createService`：兼容 service 适配入口；长期执行生命周期应由 executor registry 承担。
- `getArtifact` / `buildCacheParams`：artifact 读取和缓存参数构造。

不得暴露 `skillPath`、函数引用、service 实例、本地路径或 executor 内部细节到公开 API。

## 新增模块标准

新增模块时优先新增 descriptor，并确认：

- 是否真的需要独立 artifact、cache、重跑、trace 或独立 executor。
- 是否能明确输入、输出、依赖和 parent artifact。
- 是否有 `stage.start / stage.end / stage.fail` 和 DebugSnapshot。
- 是否保留历史，不静默覆盖核心结果。
- 是否只向前端暴露安全 metadata。
- workflow 是否只依赖 descriptor，而不是写死模块实现。
