# 功能槽位原子化产物契约

## 定位

`function-slot-atomization-analysis` 是脚本段落、节奏结构、包装结构之后的可重组结构层 artifact。

它不替代三份上游分析，也不直接承担数据库检索、前端编辑或新视频重组。它只负责把三份分析结果对齐到一条功能槽位链，并记录每个槽位中的脚本原子、节奏原子、包装原子及绑定关系。

一句话边界：

```text
script-segments + rhythm-structure + packaging-structure
-> function-slot-atomization-analysis
-> 后续检索、展示、重组能力
```

## 产物职责

该 artifact 必须回答：

- 这条样例视频由哪些功能槽位组成。
- 每个槽位让观众从什么状态变成什么状态。
- 每个槽位分别挂了哪些脚本、节奏、包装原子。
- 三类原子之间有哪些同步、依赖、替换、冲突或前后承接关系。
- 后续重组时哪些结构应保留，哪些实现可以替换。

## 生成方式

该产物应由独立分析 role/skill 生成，而不是由一次性自动脚本直接拼装。

建议 role：

- role：`function-slot-atomization-analyzer`
- skill：`function-slot-atomization-analyzer`
- 输入：三份已完成 final 结果，即 `script-segments.final.txt`、`rhythm-structure.final.txt`、`packaging-structure.final.txt`
- 输出：`function-slot-atomization-analysis` artifact

该 role 的核心任务是从三份 final 里抽象功能槽位、三类原子和绑定关系。它不重新切 shot，不重做脚本段落、节奏或包装分析，也不从原始视频素材重新推理。

`schema` 只用于约束 role 的输出结构，不代表生成方式应转成纯规则脚本。

## 不做的事

第一版产物不做：

- 不自动生成新脚本。
- 不生成新视频结构。
- 不写入关系型数据库。
- 不承担前端编辑状态。
- 不覆盖上游脚本、节奏、包装分析结果。
- 不把 shot 当作唯一原子单位。
- 不用本地规则脚本替代分析 role 的语义抽象。

## 功能槽位

`slotChain` 是该产物的主输出。一个样例通常产出多个槽位，并按视频推进顺序排列。

```json
{
  "slotId": "slot_001",
  "slotOrder": 1,
  "slotName": "痛点激活槽",
  "slotType": "problem_activation",
  "viewerStateBefore": "无感路过",
  "viewerStateAfter": "看见具体问题，并知道产品处理对象",
  "persuasionTask": "把高痛点对象推到第一视线，并立刻给出解决动作",
  "scriptAtomIds": ["script_atom_001"],
  "rhythmAtomIds": ["rhythm_atom_001"],
  "packagingAtomIds": ["packaging_atom_001"],
  "bindingIds": ["binding_001"],
  "sourceRefs": {
    "scriptSegmentLabels": ["问题区直冲开场"],
    "rhythmSectionLabels": ["贴脸连击起手"],
    "packagingBlockLabels": ["局部圈定加动作示范开场"],
    "shotRefs": ["shot_1", "shot_2", "shot_3", "shot_4"]
  },
  "confidence": 0.9,
  "needReview": false
}
```

## 原子

原子不是原文片段，也不是固定 shot。原子是某个槽位中可迁移、可替换的功能实现。

```json
{
  "atomId": "script_atom_001",
  "atomType": "script",
  "slotId": "slot_001",
  "atomName": "问题区直冲开场",
  "atomFunction": "指出问题区并引出产品动作",
  "mustKeep": ["具体问题对象", "直接解决动作"],
  "replaceableVariables": ["问题区域", "产品动作", "口播表达"],
  "avoid": ["复杂机制解释", "品牌故事前置"],
  "sourceRefs": {
    "shotRefs": ["shot_1", "shot_2", "shot_3", "shot_4"]
  },
  "confidence": 0.9
}
```

`atomType` 只允许：

- `script`
- `rhythm`
- `packaging`

## 绑定关系

`bindings` 描述槽位内或槽位之间的约束。第一版至少支持以下类型：

- `support`：一个原子增强另一个原子的表达。
- `require`：某类主张必须配某类证明或包装。
- `sync`：脚本事件、节奏点、包装事件需要同步。
- `substitute`：保留功能，允许替换表面载体。
- `conflict`：某些脚本、节奏、包装组合不应同时出现。
- `carryover`：前后槽位共享同一对象、证据或注意力线索。

```json
{
  "bindingId": "binding_001",
  "bindingType": "sync",
  "slotIds": ["slot_001"],
  "atomIds": ["script_atom_001", "rhythm_atom_001", "packaging_atom_001"],
  "rule": "问题对象出现、手指指向、产品动作落下要踩在同一注意力拍点",
  "riskIfBroken": "观众看见动作但没有建立具体痛点对象",
  "confidence": 0.9
}
```

## 重组规则

`recombinationRules` 只记录从样例中抽出的可迁移约束，不直接生成新内容。

```json
{
  "ruleId": "rule_001",
  "ruleType": "must_keep",
  "appliesTo": {
    "slotTypes": ["problem_activation"]
  },
  "rule": "开头如果使用问题区直冲，结果槽位必须回到同一问题对象或同一观众关切",
  "sourceBindingIds": ["binding_001"],
  "confidence": 0.9
}
```

## 追踪要求

该产物属于核心结果，必须遵守 `Debug追踪规范.md`：

- 必须有 `artifactId`、`parentArtifactIds`、`type`、`schemaVersion`。
- 必须保留三份上游分析 artifact 的来源。
- 主链路必须携带 `runId / traceId / stageId / artifactId / parentArtifactId`。
- stage 名称建议使用 `function_slot_atomization.analyze`。
- 失败时必须生成受控 DebugSnapshot。
- 普通日志只记录摘要，不写完整 prompt、完整素材或敏感内容。

## 后续接入边界

后续如果接入正式流程，应作为独立模块接入模块注册体系：

- `moduleId`: `function-slot-atomization`
- `artifactType`: `function-slot-atomization-analysis`
- 上游依赖：`script-segments`、`rhythm-structure`、`packaging-structure`
- 输出 artifact key：待模块接入时再定

数据库、前端和重组器只能消费该 artifact，不应重新解释三份上游分析来绕过槽位层。

## 数据库投影边界

功能槽位原子化结果可以写入本地 Projection DB 以支持检索和调试定位，但 artifact JSON 仍是唯一事实源。Projection DB 是可删除、可按单个 artifact 重投影的派生索引，不参与覆盖、返工、版本真相判断，也不替代 artifact store。

第一版投影规则：

- 默认投影库位于 `Runtime/Projection/function-slot-projection.sqlite`，该目录不入库。
- `function_slots` 必须保存 `viewerStateBefore`、`viewerStateAfter` 和 `persuasionTask`，因为功能槽位的核心定义是观众状态跃迁与说服任务。
- 三类原子使用公共身份表加 subtype 表承接差异字段，不把 script、rhythm、packaging 的可检索字段强行塞进一张宽表。
- `source_refs` 只服务溯源和调试定位，不服务业务结构重建；补投影必须从完整 artifact JSON 回放。
- `globalRiskIfBroken` 只表示 binding 被破坏时的一般风险。V1 不表达“由哪个 slot/atom 打破、在哪个上下文打破”的差异；如需该能力，后续新增上下文风险表，不改写当前 binding 语义。
- Projection 写入失败不得回滚已成功 materialize 的 artifact；必须记录结构化失败日志和受控 DebugSnapshot，并允许后续按单样例或单 artifact 重新投影。

## FunctionSlotLibrary 边界

`Artifacts/FunctionSlotLibrary/<artifactId>/` 是功能槽位原子化结果的可入库结构素材库，用于 diff、review、分享和从精选结构重新写入本地 Projection DB。它不是事实源，也不是重组器运行状态库。

第一版 Library 规则：

- 导出只从 `Runtime/Artifacts/<sampleVideoId>/artifact.json` 中完整的 `functionSlotAtomizationAnalysis` 读取，不接受裸 Agent JSON 直接入库。
- 每个 library item 固定拆成 `manifest.json`、`slots.json`、三类 `atoms.*.json`、`bindings.json`、`rules.json` 和 `templates.json`。
- 导出内容只保存 normalized 结构，不保存完整 prompt、DebugSnapshot、inputPackage、视频、抽帧或本地绝对路径。
- `manifest.json` 保存 `artifactId / sampleVideoId / traceId / parentArtifactId / source*ArtifactId / counts / contentHash`，用于审阅和追踪。
- 从 Library 投影到 SQLite 时，只投影指定 `artifactId`，不读取 Runtime artifact，也不清空全库。
- 删除 Library item 只删除 `Artifacts/FunctionSlotLibrary/<artifactId>/`，不级联删除 Projection DB 中已有投影。

它和现有几层的区别：

- `Runtime/Artifacts/<sampleVideoId>/artifact.json`：事实源，记录当前样例完整产物和血缘。
- `Artifacts/FunctionSlotLibrary/<artifactId>/`：可被 Git 跟踪的精选结构素材库，服务 review、diff 和可控导入。
- `Runtime/Projection/function-slot-projection.sqlite`：本地派生查询索引，可删除、可按单 artifact 重投影，不入库。
- `Artifacts/AnalysisFinalOutputs`：latest 文本导出层，便于人工查看阶段 final output，不承载功能槽位结构素材库职责。
