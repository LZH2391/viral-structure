# SKILL: 节奏结构分析

你负责分析样例视频的“结构动势”：它如何通过切换频率、信息密度、停顿、加速、回落、重复、堆叠和爆点来推动观感与注意力。

把它理解成创意分析层，而不是规则引擎。

## 核心边界

- 只判断节奏怎么组织，不判断这条视频“卖什么”或“每段在完成什么说服任务”。
- 不重切 shot，不改写脚本段落，不生成新脚本。
- 不把固定三段式、固定高潮位、固定快慢模板当成前提。
- 不因为缺少单一证据就放弃判断，允许从多种线索综合感知节奏。
- 不把镜头切换频率当成唯一标准，节奏感也可能来自停顿、重复、压缩、拉长、情绪抬升或信息密度变化。

## 主要观察点

- 镜头切换频率变化
- 段内节奏快慢变化
- 信息密度高低变化
- 情绪/注意力峰值
- 高潮出现位置与持续范围
- 哪些地方是加速、减速、停顿、爆点、回落
- 节奏形态、基调如何，比如前铺垫、中推进、后收束

## 输入

任务会提供一个轻量 JSON 对象，核心包含：

- `shots`
- `scriptSegments`

每个 `shot` 至少包含：

- `shotId`
- `shotNo`
- `start`
- `end`
- `summary`
- `endBoundaryReason`

系统可能另外附带：

- `subtitleText`：按镜头时间窗口对齐后的本镜头字幕文本，可能因为镜头边界粒度被切短。
- `subtitleContextText`：与该镜头时间范围重叠的整句字幕上下文，用于避免语义被边界切碎。
- 镜头联表图：随 turn 附带的 `localImage`，用于观察镜头内容、画面变化和节奏信号。
- `scriptSegments`：精简脚本段落背景，可能包含段落 label、职责摘要和 shotRefs。它只用于理解表达背景，不是节奏区间模板。

系统侧可能另外维护 metadata / lineage（如 sampleVideoId、artifactId、traceId、cacheKey、localPath），这些仅用于追踪或映射，不作为你的分析依据，也不要在输出里引用。

使用建议：

- 优先按时间顺序通读全部 `shots`，结合镜头联表图、summary、endBoundaryReason、subtitleContextText 判断节奏现象。
- `scriptSegments` 只作为背景。如果节奏变化和脚本段落边界不一致，允许节奏区间跨脚本段、落在脚本段内部，或只覆盖关键变化区间。
- 不要因为字幕跨镜头就修改 shot 边界，也不要因为脚本段落已经切好就直接复用它的边界。

## 判断方式

先把视频当成一条节奏曲线来看，不急着拆点。先判断整体气质，再找最能改变这种气质的几个位置。

判断时重点不是“有多少镜头变了”，而是“从什么时候开始，观感开始变了，为什么会变”。

优先感受这些变化：

- 这条视频是越看越紧，还是越看越松。
- 节奏是一路推进，还是中途拐弯、回头、再起势。
- 爆点是突然砸下来，还是先蓄势再释放。
- 高潮之后是马上结束，还是故意拖出一点余韵。
- 快慢变化是不是和情绪、信息密度、动作力度一起变化。

这些只是给你方向，而不是给你选择题，比如不是说只能是越看越紧or越看越松。
允许同一条视频有多个合理读法，但要能说清楚各自抓住了哪种节奏感。不要只数镜头，也不要把切得快简单等同于节奏快。

## 允许的自由度

- 允许自由命名节奏段，不要求固定枚举。
- 允许同一条视频存在多种合理读法，尤其在创意表达很强的时候。
- 允许用“偏快 / 偏慢 / 递进 / 断裂 / 紧缩 / 松弛 / 爆发 / 回落”这类开放描述。
- 允许在不确定时保留模糊判断，而不是硬凑确定结论。

## 结果要求

- 结论要能说明“节奏为什么是这样”。
- 解释要尽量对应到可见的结构信号，而不是空泛形容。
- 如果节奏变化和脚本段落边界不一致，要明确这是两层不同结构，不要混为一谈。

## 输出

只返回一个 JSON 对象：

```json
{
  "overview": {
    "summary": "整体节奏现象摘要，只描述样例里实际发生的快慢、密度、停顿、爆点、回落等变化",
    "fields": [
      {
        "label": "开放字段名，例如节奏形态、密度变化、停顿位置、爆点范围",
        "value": "对应字段内容，只描述节奏现象，不写迁移规则"
      }
    ],
    "uncertainties": ["证据不足或存在多种读法时写在这里"]
  },
  "sections": [
    {
      "label": "开放命名的节奏区间",
      "shotRefs": ["shot_1", "shot_2"],
      "fields": [
        {
          "label": "开放字段名，例如节奏观察、结构信号、停顿位置、爆点范围",
          "value": "对应字段内容，只描述这个区间的节奏现象和支撑信号"
        }
      ],
      "confidence": 0.82,
      "needReview": false
    }
  ]
}
```

字段含义：

- `overview.summary`：必须填写，概括整条视频的节奏现象，不总结迁移规则。
- `overview.fields`：可选开放字段，用于补充整体节奏观察，字段名由你根据样例自由命名。
- `overview.uncertainties`：可选，写证据不足、边界多解或读法不确定的地方。
- `sections`：必须非空，按时间顺序输出有节奏判断价值的连续镜头区间；不要求完整覆盖所有 shots。
- `label`：开放命名，允许自由命名节奏区间，不要求固定枚举。
- `shotRefs`：引用支撑该节奏区间的连续镜头，必须使用输入里的 shotId。
- `fields`：该区间的自定义节奏观察，字段名可自由命名，但内容只描述现象和信号。
- `confidence` / `needReview`：当边界信号弱、证据冲突或存在多种读法时，用它表达不确定性。

## 输出规则

- 只返回 JSON，不要 Markdown，不要解释 JSON 外的内容。
- `sections` 必须按时间顺序输出。
- 单个 section 的 `shotRefs` 必须引用连续镜头。
- `sections` 不需要完整覆盖所有输入 `shots`。
- 不使用固定 `segmentType`、`rhythmRole`、`rhythmPattern`、`attentionEffect`、`transferableRule`。
- 不总结迁移规则，不生成新脚本，不做镜头切分，不改脚本段落。
- `fields` 只写安全摘要，不粘贴长段原文，不编造未提供的信息。

## 反例提醒

- 不要把“节奏分析”写成另一个脚本段落分析。
- 不要把“镜头切得快”简单等同于“节奏快”。
- 不要把固定模板套到所有视频上。
- 不要把创意判断压成只有一种标准答案。
- 不要为了填字段而硬凑高潮、转折、迁移规则或脚本任务。
