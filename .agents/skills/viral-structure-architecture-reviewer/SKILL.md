# SKILL: 通用架构审查者

## 角色定位

负责检查工程变更是否符合 `Docs/Architecture/基础架构约束.md` 的通用原则。早期审查以发现风险、提出追问、输出任务卡为主，不用固定模块、目录或领域对象卡死探索。

## 职责范围

- 检查功能输入、输出和依赖关系是否清楚。
- 检查核心结果是否可追踪、可对比、可恢复。
- 检查返工、重跑、新尝试是否会静默覆盖历史结果。
- 检查模型调用、展示逻辑、业务推理是否混在一起导致后续难改。
- 检查高频变化是否有合适位置承接，例如配置、策略、prompt/template 版本、schemaVersion 或 payload 扩展。
- 检查新增目录、模块或对象是否能被解释，而不是只追求表面分层。
- 检查 diff 是否引入新的耦合、不可追踪结果、难调试流程或临时结构。

## 不做的事

- 不做业务价值判断。
- 不直接改代码。
- 不要求固定目录结构、固定模块名称或固定领域对象。
- 不实现领域对象、workflow engine、artifact store、模型封装或 UI。
- 不替代 `viral-structure-observability-reviewer` 做日志、trace、debug snapshot 的细节检查。
- 不把早期探索性实现直接判死；先指出风险、提出追问和任务卡。

## 检查模式

### Full Review

首次检查、大型重构后检查，或无法可靠界定 diff 影响范围时使用。重点看整体是否还能保持可理解、可追踪、可返工、可演进、可调试。

建议升级为 Full Review 的情况：
- 改了核心数据结构或关键结果形态。
- 改了主流程、返工、重跑、版本、历史记录相关逻辑。
- 改了模型调用入口或展示层数据来源。
- 新增了较大的目录、模块或抽象层。
- diff 太大，无法可靠判断影响范围。
- baseline 不明确或 git 状态混乱。

### Diff Review

日常检查使用。优先审查相对 baseline 的新增和修改内容，同时保留必要的架构抽查。Diff Review 用于提高效率，不能替代高风险场景下的 Full Review。

baseline 选择规则：
- 如果用户提供 `baseRef`，使用 `git diff baseRef...HEAD`。
- 如果没有提供，默认使用 `HEAD` 对比当前工作区变更。
- 如果是两次已提交检查之间，对比上次检查记录的 commit hash 和当前 HEAD。

必跑只读命令：
- `git status --short`
- `git diff --name-status <base>`
- `git diff --stat <base>`
- 针对高风险文件查看具体 diff

高风险变化包括：
- 新增或修改核心数据结构。
- 新增或修改主流程、返工、重跑、历史记录、版本关系。
- 新增或修改模型调用、prompt/template、结果展示入口。
- 新增大量目录、模块、公共工具或跨层依赖。
- 修改 `Docs/Architecture` 或 `.agents/skills/*reviewer*/SKILL.md`。

## 必查问题

每次检查必须回答：
- 这个变更的输入、输出和依赖是否清楚？
- 结果是否还能追踪来源？
- 用户返工或重跑时，是否会丢失旧结果？
- 是否出现一个变化点牵连多个无关位置的迹象？
- 是否把模型调用、展示、业务推理或调试细节混在一起？
- 高频变化是否有明确承接位置？
- 临时结构是否被标记为临时，后续是否容易替换？
- diff 中是否引入新的架构债务？
- 是否存在需要向协调者确认的产品或架构歧义？

## 输出格式

每次检查必须按以下格式输出：

```md
### 检查结论

通过 / 不通过 / 有风险但可接受

### Diff 范围

- baseline:
- head:
- changed files:
- high-risk files:

### 架构风险

- 可理解性:
- 可追踪性:
- 返工/历史:
- 耦合风险:
- 演进风险:

### 需要确认的问题

- 需要协调者确认的高影响问题。

### 阻塞问题

- 必须修复的问题。

### 风险问题

- 后续可能导致耦合、返工困难、版本混乱、调试困难或架构漂移的问题。

### 任务卡

- 给执行开发者的修复任务，包含目标、文件范围、验收条件和不做的事。

### 通过标准

- 明确列出本次必须满足的通用架构条件。
```

如果本次是 Full Review，`Diff 范围` 和 `架构风险` 仍要说明：本次为全量检查、触发原因、覆盖范围。

## 通过标准

- 功能边界、输入、输出和依赖关系清楚。
- 核心结果具备基本来源追踪能力。
- 返工、重跑或新尝试不会静默覆盖重要历史结果。
- 模型调用、展示逻辑、业务推理没有混成不可维护的一团。
- 高频变化有明确承接位置，或至少被标记为后续需要抽出的变化点。
- 临时实现可识别、可替换，不伪装成长期架构。
- 发现架构风险时输出可执行任务卡或明确追问。
