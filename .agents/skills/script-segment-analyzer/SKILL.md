# SKILL: 脚本段落分析

你只分析任务中提供的 `shots[]`、`commerceBrief`，以及显式附带的字幕/OCR/音频摘要。不要读取无关文件，不要做结构迁移，不要生成新内容。

你的目标不是把输入机械分组，而是从连续镜头里归纳样例视频的“脚本/段落结构”: 这条带货表达在按什么说服任务往前推进，哪里发生了任务切换，为什么在这里切段。

## 核心认识

- `段落` 不是镜头组，也不是固定三段或固定模板。
- 一个段落是一串连续镜头共同完成的同一个表达任务，例如先抛问题、再给结果、再做证明、再推进转化。
- 段落边界来自“表达任务变化”，不是来自镜头数量、时长均分、画面切换频率，或你对常见短视频结构的先验套模板。
- 同一段里可以有多个镜头，只要它们仍在服务同一个说服任务，就应该归在一起。
- 相反，即使镜头数量不多，只要表达任务已经明显切换，就应该切出新段。

## 输入
任务会提供一个轻量 JSON 对象，包含：

- sampleVideoId
- sourceArtifactId
- shots
- commerceBrief

每个 `shot` 至少包含：

- shotId
- start
- end
- summary

可选辅助信息：

- subtitleSummary
- ocrSummary
- audioCueSummary

## 怎么判断段落边界

先按时间顺序通读全部 `shots`，理解这条带货视频的说服链路，再决定如何切段。判断时以“当前这几个镜头正在完成什么表达任务”为主轴。

优先关注这些边界信号：

- 从提出问题转为展示结果。
- 从展示产品转为证明效果。
- 从单点卖点转为多证据堆叠。
- 从证明转为价格、优惠或行动引导。
- 从情绪刺激转为理性解释。
- 从场景铺垫转为使用演示。

也可以出现其他任务切换，只要你能从镜头摘要、字幕、OCR、口播摘要中说明证据，就可以切段。

以下情况通常不应该单独作为切段依据：

- 只是换了机位、景别、构图，但表达任务没变。
- 只是同一结论换了一个补充镜头。
- 只是同一证明链路里的证据继续堆叠，没有切到新的说服任务。
- 因为“短视频常见三段式”就默认切成三段。

## 对带货样例的理解框架

你可以借助下面这些观察维度理解样例，但它们只是分析视角，不是固定段落类型枚举：

- 痛点或欲望
- 产品出现
- 效果承诺
- 证明方式
- 信任增强
- 转化动作

不要强行要求每条样例都完整覆盖这些维度，也不要把这些维度直接当作固定 label 列表。

## 输出时每段必须回答的问题

对每一个 `segment`，你都要先在心里回答四个问题，再填写 JSON：

1. 这一段在说服链路里承担什么具体任务。
2. 它依赖哪些镜头、字幕、OCR 或口播证据。
3. 它如何承接上一段，并把表达推进到下一段。
4. 它可迁移的规则是什么。

字段含义要这样落实：

- `label`：开放命名，允许自由命名，例如“痛点钩子”“效果抛出”“实拍证明”“信任补强”“下单收束”。不要要求固定枚举，也不要只写“第一段/第二段”。
- `roleInScript`：写清这段在样例脚本中的具体说服职责，并点明它如何承接上一段、导向下一段。不要只写“承接表达”“过渡”这类空话。
- `shotRefs`：引用支撑这段任务的连续镜头，必须按时间顺序完整覆盖输入 `shots`，不要漏镜头，也不要无意义重复到相邻段。
- `evidence`：只写支持你判断的安全摘要，说明你为什么认为这些镜头在完成这个任务。
- `transferableRule`：总结“这类任务通常如何被组织出来”的可迁移规则，抽象的是结构方法，不是新商品脚本。
- `confidence` / `needReview`：当边界信号弱、证据冲突或一组镜头可能兼具两种任务时，用它表达不确定性。

## 输出
只返回一个 JSON 对象：

```json
{
  "segments": [
    {
      "label": "这一段叫什么",
      "roleInScript": "它承担什么具体说服任务，并如何承接上一段、导向下一段",
      "shotRefs": ["shot_1", "shot_2"],
      "evidence": ["支撑判断的画面/字幕/口播证据"],
      "transferableRule": "这段结构能迁移的规则",
      "confidence": 0.82,
      "needReview": false
    }
  ]
}
```

## 规则
- `segments` 必须按时间顺序输出。
- `segments` 必须完整且按顺序覆盖所有输入 `shots`。
- `shotRefs` 必须引用输入里的 shotId，且同一镜头不要跨多个相邻段无意义重复。
- 用“表达任务变化”切段，不要默认固定三段，也不要把每个镜头都拆成独立段落。
- `label` 使用开放标签，不要求固定枚举。
- `roleInScript` 描述该段在样例脚本中的具体说服职责，不要发散到新内容创作，也不要写成空泛的“承接表达”。
- `evidence` 只写安全摘要，不粘贴长段原文，不编造未提供的信息。
- `transferableRule` 只总结结构规则，不生成新脚本。
- 不使用固定 `segmentType` 枚举。
- 不做镜头切分，不改写 `commerceBrief`，不生成结构迁移方案。

## 反例提醒

以下做法都算偏题：

- 把段落理解成“若干相邻镜头的分组结果”，却不说明说服任务为何变化。
- 先假设视频一定有三段，再回头给三段找理由。
- 把 `commerceBrief` 里的词直接抄成段落类型，忽略实际镜头证据。
- 把 `roleInScript` 写成泛泛的“介绍产品”“承接上文”“总结收尾”。
- 在 `transferableRule` 里开始替新商品写新话术、新剧情或新品类脚本。
