## 仓库工作约定

### Git 处理
- 默认不自动提交 `commit`，由用户手动提交。
- 每完成一轮可交付的变更后，提醒用户需要手动提交，并给出本次相关文件、验证结果、和建议 commit message。
- 建议 commit message基于你自己的改动，不要浪费时间去看git stauts,git diff，遗留什么的。
- 只有在用户明确要求“提交 / commit / 直接提交”时，才执行提交。

### 中间产物
- 中间产物、临时文件、生成结果优先加入 `.gitignore`。
- 已经产生但不应入库的文件，及时清理或补充忽略规则。

### 编码约定
- 读取文件或命令文本输出时，默认显式使用 UTF-8，避免乱码。

### 架构约束
- 通用架构原则以 `Docs/Architecture/基础架构约束.md` 为准；阶段计划和讨论结果只作为本地材料，不进入公开仓库。
- 新增核心能力、重构、返工流程、模型链路或数据流时，必须先对照通用架构原则，避免不可追踪、难返工和难演进。

### 测试策略
- 优先运行与本次改动直接相关的定向测试。
- 除非用户显示要求，不得进行全量测试，你觉得改动已经很大需要了，可以建议提醒。

### 日志与调试硬约束
- Debug 追踪通用规则以 `Docs/Architecture/Debug追踪规范.md` 为准。
- 主链路必须携带 `runId / traceId / stageId / artifactId / parentArtifactId`，保证结果能追溯到上游产物。
- 核心阶段必须有结构化开始、结束、失败日志，并生成受控 `DebugSnapshot`。
- 正式代码禁止保留裸 `console.log / print / 临时 dump`，调试产物必须清理或加入 `.gitignore`。
- 模型调用必须记录 stage、prompt/template 版本、输入摘要、输出摘要、解析结果、失败与重试信息。
- 返工、分支、重跑必须保留来源 artifact，以及新旧版本关系。
- 普通日志不得裸写敏感内容；完整内容只能进入受控调试快照。
- 新功能缺少日志、trace、debug snapshot，视为未完成。

### 例外说明
- 只有在任务明确要求时，才提交配置文件、工作区文件或其他本地环境产物。
